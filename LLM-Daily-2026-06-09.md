JetBrains released **Mellum2**, a 12B-parameter Mixture-of-Experts model purpose-built for software engineering workflows — code generation, natural language routing, and agent orchestration. Released under Apache 2.0, it's the first "focal model" from a major developer tool vendor: a narrow, fast component designed to replace frontier-model calls for high-frequency, latency-sensitive tasks like query routing, summarization, and sub-agent coordination.

- **Architecture:** 12B total params, 2.5B active per token (64 experts, 8 active), 131K context window, bfloat16 precision

**Why it matters:** Mellum2 represents a new category — "focal models" — optimized for the high-frequency, low-latency tasks that dominate agent workflows. Pre-training used the Muon optimizer under FP8 hybrid precision over ~10.6T tokens through a three-phase curriculum shifting from diverse web data toward curated code and math content.

### 1b: NVIDIA Nemotron 3 Ultra — Deep-Dive Verification
**Signal:** `#SIGNAL` (Updated) | **Date:** June 8, 2026 | **Sources:** developer.nvidia.com/blog, maximelabonne.substack.com/p/nemotron-3-ultra-what-distillation
**Confidence:** 95%

Continuing from the June 4/8 digest: independent technical analysis by Maxime Labonne confirms and extends the NVIDIA blog claims with critical new details about training instability.

**Key finding — Training divergence:** Ultra trained on only 20T tokens, fewer than planned (Nano and Super both used 25T). The run diverged twice:
- Around 8T tokens: a throughput tweak moved the output layer's gradient accumulation to BF16, which "quietly zeroed out the MTP heads' contribution" — the MTP loss spiked before the main loss showed anything. Reverting to FP32 fixed it.
- Around 16T tokens: a second divergence NVIDIA never root-caused. They annealed the learning rate early and stopped at 20T instead of the planned horizon, linking it only by correlation to "dead experts and a ballooning residual stream."

**What this means:** NVFP4 pretraining at this scale is still experimental. The training instability cost Ultra an estimated 20% of its planned data budget. This is a genuine risk signal for anyone considering fine-tuning or distilling from the checkpoint — the model's training curve has a known gap.

**Evidence:** T3 (NVIDIA official blog deep-dive). T2 (Labonne analysis confirming MOPD method, divergence timeline, and LatentMoE architecture details).

Claude Code shipped v2.1.166–168 (June 6) with significant agent reliability improvements:

- **`--safe-mode` flag:** Starts Claude Code with all customizations (CLAUDE.md, plugins, skills, hooks, MCP servers) disabled — critical for troubleshooting agent behavior
- **`/cd` command:** Change working directory mid-session without breaking the prompt cache
- **`fallbackModel` setting:** Configure up to 3 fallback models tried in sequence when the primary model is overloaded or unavailable; `--fallback-model` now applies to interactive sessions

- **Cross-session hardening:** Messages relayed via SendMessage from other Claude sessions no longer carry user authority — receivers refuse relayed permission requests, auto mode blocks them

**Karl-relevance:** HIGH. The `fallbackModel` setting is directly applicable to OpenClaw's agent reliability — if Claude Code API goes down, you can fail over to Sonnet or another model.
