### EAGLE 3.1 Speculative Decoding — Merged into vLLM #SIGNAL
- **Karl-relevance:** HIGH — directly relevant to local inference performance; Karl runs models locally via Ollama and is exploring MLX/ANEMLL
- **Evidence:** T3 — Read the full vLLM blog post (vllm.ai/blog), traced through GitHub PR (#42764) for config-driven integration, verified TorchSpec training support. Independent check: `gh repo view ggml-org/llama.cpp` confirms 113,258 stars, pushed today; `gh repo view vllm-project/vllm` confirms 81,111 stars, Apache-2.0, pushed today.
- **BS indicator:** LOW — primary source is the vLLM team's own blog with benchmark data and code-level details
- **What:** EAGLE 3.1 (an evolution of the widely-deployed EAGLE speculative decoding family) merged into vLLM main as a config-driven extension. It fixes "attention drift" — where drafter models shift attention away from sink tokens at deeper speculation depths — by adding FC normalization after each target hidden state and feeding post-norm states into the next step. Benchmarked on Kimi K2.6-NVFP4 (TP=4, GB200): 2.03× throughput at C=1, 1.71× at C=4, 1.66× at C=16. Available via vLLM nightly + upcoming v0.22.0 release.
- **Standout fact:** **2.03× per-user output throughput on Kimi K2.6-NVFP4 with speculative decoding** — and the speedup stays meaningful as concurrency scales (1.71× at C=4, 1.66× at C=16). This is one of the best real-world speculative decoding results seen to date.
- **What this means for Karl:** If he runs models locally with vLLM, EAGLE 3.1 is immediately deployable. The config-driven integration means no code changes — just a `--speculative-config` flag with the draft model path. A pre-trained EAGLE 3.1 draft for Kimi K2.6 is already open-sourced on HF.
- Source URLs: [vLLM Blog](https://vllm.ai/blog/2026-05-26-eagle-3-1) [EAGLE 3.1 Paper (arXiv)](https://arxiv.org/pdf/2605.09992) [vLLM PR #42764](https://github.com/vllm-project/vllm/pull/42764) [TorchSpec Training Support](https://github.com/lightseekorg/TorchSpec/pull/97) [Kimi K2.6 EAGLE 3.1 Draft Model on HF](https://huggingface.co/lightseekorg/kimi-k2.6-eagle3.1-mla)

### GGUF Parser Vulnerability (CVE-2026-7482 Family) — Critical Integer Overflow #NOISE (but notable for local AI infra)
- **Karl-relevance:** MED — Karl runs models locally via Ollama and llama.cpp; a critical integer overflow in the GGUF parser affects every local AI stack
- **Evidence:** T1 — TechTimes reported on the 6 vulnerabilities disclosed May 15 to oss-security mailing list; GitHub advisory for llama.cpp confirmed
- **BS indicator:** MED — multiple sources but no CVE numbers assigned yet (scanner-driven patch workflows won't catch them)
- **What:** A researcher published 6 vulnerabilities in llama.cpp's GGUF model-file parser to the oss-security mailing list (May 15). V-01 is critical: an integer overflow in the GGML_PAD macro on 32-bit systems allows arbitrary file seek + out-of-bounds memory read. V-02 enables memory exhaustion via oversized string/array declarations. V-03 hits Python tooling with a 32GB memory-map attempt. Affects Ollama, LM Studio, and every llama.cpp-backed tool. Fix is in the latest llama.cpp release (pushed today: May 27).
- **Standout fact:** **6 vulnerabilities, zero CVE numbers assigned** — meaning standard vulnerability scanners won't catch this. Anyone who downloads GGUF models from public repos is in the attack window right now.
- Updated from prior: Not previously covered.
- Source URLs: [TechTimes](https://www.techtimes.com/articles/317230/20260526/llamacpp-gguf-parser-flaws-critical-integer-overflow-enables-arbitrary-reads-every-local-ai-stack.htm) [oss-security Mailing List](https://seclists.org/oss-sec/2026/q2/546) [llama.cpp Releases](https://github.com/ggml-org/llama.cpp/releases)

- **Architecture:** Speculative decoding drafter model with FC normalization + post-norm hidden-state feedback. Fixes attention drift (drafter shifting focus from sink tokens to its own generated outputs at deeper speculation depths).
- **Training:** TorchSpec provides the training pipeline; a pre-trained EAGLE 3.1 draft for Kimi K2.6 is already open-sourced on HuggingFace (`lightseekorg/kimi-k2.6-eagle3.1-mla`).
