### F03: Unsloth NVFP4 Quants for Qwen3.6 — 2.5× Faster, Zero Accuracy Loss
**Date:** July 10, 2026
**Signal:** `#SIGNAL`
**Evidence Tier:** T2 | **Confidence:** 75%
**Sources:**
- [Unsloth Documentation](https://unsloth.ai/docs/models/qwen3.6)
- [DawnSift Community Report](https://dawnsift.com/t/ai)

Unsloth released new **dynamic NVFP4 quantization for Qwen3.6** (27B and 35B-A3B):
- **2.5× faster** than NVIDIA's baseline NVFP4 quants on the 27B model
- **1.56–1.79× faster** on the 35B-A3B variant
- **No accuracy degradation** — verified against baseline
- Qwen3.6-27B at Q4 runs on **18GB RAM** (15GB for 3-bit)
- Qwen3.6-35B-A3B at Q4 runs on **23GB RAM** (17GB for 3-bit)
- Also supports MTP (Multi-Token Prediction) for 1.4–2.2× faster inference

Hardware table from Unsloth docs:

|-------|-------|-------|-------|-------|------|
| 27B   | 15 GB | 18 GB | 24 GB | 30 GB | 55 GB |
| 35B-A3B | 17 GB | 23 GB | 30 GB | 38 GB | 70 GB |

**Why it matters:** For Karl's M4 Max (128GB unified memory), this means both Qwen3.6 variants fit comfortably in RAM with headroom. The 2.5× speedup on NVFP4 is a significant inference acceleration — especially for agentic coding workloads where token throughput directly impacts session latency.

### F04: oMLX — SSD-Persistent KV Cache LLM Server for Apple Silicon
**Date:** July 2026 (new repo)
**Signal:** `#SIGNAL`
**Evidence Tier:** T2 | **Confidence:** 70%
**Sources:**
- [GitHub: jundot/omlx](https://github.com/jundot/omlx)
- [MiraHeise AI Wiki](https://ai.miraheze.org/wiki/Mtplx)

New open-source project: **oMLX**, an LLM inference server for Apple Silicon managed from the macOS menu bar. Key features:
- **Continuous batching** with per-model settings and a live dashboard
- **Tiered KV cache**: hot in-memory tier + cold SSD tier — sessions persist to disk across app restarts, allowing near-instant restoration of conversation state
- Supports text LLMs, VLMs, OCR models, embeddings, and rerankers
- OpenAI-compatible API at `http://localhost:8000/v1`
- Apache 2.0 licensed
- macOS 15.0+ (Sequoia), Python 3.11–3.13, Apple Silicon only
- Available via `.dmg` download or Homebrew (`brew tap jundot/omlx`)

**Why it matters:** This is a direct answer to the "stateless LLM" problem in Category L. The SSD-persistent KV cache means Karl's local conversations survive app restarts — a meaningful step toward persistent state on local hardware. Combined with oMLX's continuous batching, this fills a gap between Ollama (simple) and more complex inference servers.

**Why it matters:** Hermes Agent occupies an interesting position in the agent landscape. It's complementary to OpenClaw (both are persistent-memory agents) but targets a different use case: personal automation and skill-compounding rather than development orchestration. For Karl: worth watching as a potential skill source — Hermes writes its own skills, which could feed into OpenClaw's skill framework.

**The Convergence Pattern:** All five findings point toward the same trend: coding agents are becoming the central competitive battleground, open-weight models are closing the gap with API models on routine tasks, and local inference infrastructure is maturing fast enough to make "local-first" a real cost advantage. The triopoly (OpenAI/Anthropic/Meta) is fighting for the agent layer; open-source is winning the model layer.
